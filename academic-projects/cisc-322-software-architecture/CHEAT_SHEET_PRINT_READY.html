J. ARCHITECTURAL STYLES — Q2 REFERENCE

StyleComponentsConnectorsPros (when to use)Cons (when NOT)Examples + Key Facts
Repository Central store + operators Proc calls, mem access Efficient large data; centralized backup/security; many ops on shared data Must agree on schema first; hard to distribute; evolution expensive IDEs, compilers, graphical editors. Blackboard = collab variant
Pipe & Filter Filters (transforms) Pipes (streams) Easy compose/reuse; concurrent exec; throughput analysis Bad for interactive; excess parse overhead Unix shell, video. Filters DON'T share state, DON'T know neighbors
Object-Oriented Objects (data+ops) Method calls Hidden representation; modular MUST know target identity; side effects if shared OOP systems. Opposite of Pub-Sub (identity req'd)
Pub-Sub Publishers + subscribers Event bus Strong reuse (register for events); easy evolution; no identity needed No idea who responds; can't rely on order; don't know when done GUI events, plugins, DB triggers. Often mixed with proc calls
Layered Layer groups Proc calls (restricted) Abstraction levels; changes affect ≤2 layers; interchangeable impls Not all fit layers; perf may need skip layers OSI, OS, web 3-tier. Exception: can skip for efficiency
Client-Server Clients + Servers Network Data distribution; transparent location; mix platforms Network dependent; tricky design; need registry for discovery Web, email, DB. Combine with Layered for web apps
Peer-to-Peer Peers (both) Direct network No single failure point; resilient; scalable; privacy Cliques disconnect; need peer list/directory; no guaranteed response BitTorrent, blockchain. Transient, ad hoc connections
KEY DISTINCTIONS: Pub-Sub ≠ OO (identity) · P&F: filters independent, no shared state · MVC = pattern NOT style · Blackboard = collaborative Repository · Layered can skip for perf · Batch Seq = degenerate P&F (ALL in before ANY out)

K. REFERENCE ARCHITECTURES (Exam Essentials)

Ref. Arch = template for systems in a domain. Benefits: proven designs, common vocabulary, reuse

Web Server (7 subsystems, 2 layers)

Server Layer: Reception (parse HTTP) · Request Analysis (URL→file) · Access Control (auth/authz) · Resource Handler (static vs dynamic) · Transaction Log
Support Layer: Utility · OSAL (OS portability)

Linux Kernel (5 subsystems)

SCHED (most central, all depend on it) · MM (virtual memory) · VFS (Façade pattern, file interface) · NET (BSD sockets) · IPC (signals, pipes, shared mem)

Conceptual ≠ Concrete (Linux: 19 vs 37 dependencies)

Why mismatch: (1) Missing relations (undocumented) (2) Efficiency bypasses (3) Dev expediency (code in wrong place) (4) Far more concrete deps
